Reflections:
romanisation tables and versioning

Often the ALA-LC Romanization tables are closer to being transcription systems, rather than transliteration systems. They sort of sit between the two approaches.

To interpret existing romanisation tables, it is necessary to review:

  • the 1997 edition of the tables,
  • the MARC-8 to Unicode mapping,
  • changes to the mapping, and
  • relevant sections of the Unicode Standard.

Versioning

Issues

  1. BCP 47 language tags require identification of the table version.
  2. Lack of consistency in application of tables.
  3. Availability of information on which characters to use when tables are ambiguous.
  1. BCP 47 language tags require identification of version, when using the `t` extension combined with the `m0` subtag
    • ru-Latin-t-ru-Cyrl-m0-alaloc-1997
    • ru-Latin-t-ru-Cyrl-m0-alaloc-2012
    • bg-Latin-t-bg-Cyrl-m0-alaloc-1997
    • bg-Latin-t-bg-Cyrl-m0-alaloc-2011
    • bg-Latin-t-bg-Cyrl-m0-alaloc-2012
    • bg-Latin-t-bg-Cyrl-m0-alaloc-2013

  2. The key issue is the lack of consistency in the interpreting and implementation of romanisation, both between institutions and within an institution over time, i.e. cataloguers that have worked with the 1997 tables, or work within institutions whose internal guidelines are based on knowledge of 1997 tables, versus cataloguers unfamiliar with the 1997 tables and have based their cataloging on the revised tables alone.

  3. There was a fundamental change between the 1997 tables and later tables. The 1997 tables included after the table a list of extended Latin characters used in the table, providing both the MARC-8 character name and the G1 byte used. These lists were drop from subsequent tables, and no guidance on appropriate Unicode characters were provided.

Romanisation tables are not static, they change over time.

  • Changes can be fundamental, with wide ranging effects, i.e. Wade-Giles to Pinyin migration.
  • Others can be subtle, but result in divergent practices:
    • Potentially inadvertent changes, e.g. Lao and Thai romanisation (combining left half ring below vs ogonek)
    • Changes in advice on MARC-8 to Unicode mappings

Alif and ayn

  • Up to and including 1997, tables identified MARC-8 characters to use.
  • Ayn MARC-8 to Unicode mapping changed in 1999.
  • Alif MARC-8 to Unicode mapping changed in 2005.
  • Ayn and Alif are names of MARC-8 characters, not Unicode characters.
  • The references to Ayn and Alif in Spacing Modifier Letters chart refer to different Unicode characters.

Sinopia editor issue

A Sinopia editor issue illustrates misunderstandings that can occur:

							Arabic scripts
							Alif: (Modifier letter right half ring) U+02BE officially
							http://www.unicode-symbol.com/u/02BE.html
							However, OCLC says:
							"Use U+02BC (modifier letter apostrophe) to represent the alif character"
							See OCLC table at: https://www.oclc.org/bibformats/en/onlinecataloging.html#characters

Suggestions

  • Add Unicode codepoints to romanisation table.
  • PDF and doc files include year in file name.
  • Update fonts with more suitable fonts.
  • Replace images of text with corresponding text.
  • Older versions of tables also made available with versioning.
  • Add changes document for each romanisation table.
  • Consider including machine readable versions of the tables, although mapping would need to be unidirectional.
